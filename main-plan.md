# The Best Learning Resource for Each of 17 Linux/Sysadmin Fundamentals Topics

## TL;DR
- For **12 of the 17 topics** there is a resource that genuinely hits the NetworkChuck bar (analogy → real mechanism → technically accurate, ideally illustrated): Julia Evans' Wizard Zines/comics dominate the file-descriptor, permissions and networking topics; the animated **Core Dumped** channel is the best pick for processes; and author/designer primary sources (Rob Landley on the FHS, Lennart Poettering on systemd, Cloudflare + howhttps.works + xargs on TLS, Hussein Nasser on reverse proxies) cover the rest.
- **Five topics have NO single resource that fully meets the bar, and I flag each explicitly:** Signals (4), Environment variables & shell inheritance (5), Disk/inode exhaustion (8), sudo vs su (16), and - partially - Cron internals (17). For these I give the best-available option with its limitations stated, per your request for honesty over forced recommendations.
- **Overall play:** build a personal syllabus around three anchors - Julia Evans' zines (visual intuition), Core Dumped (animated mechanism), and Brian Ward's *How Linux Works, 3rd ed.* (2021, the connective-tissue reference that accurately covers processes, signals, boot, and the privilege model in one place) - then supplement with the per-topic picks below.

## Key Findings
- **Julia Evans (jvns.ca / wizardzines.com)** is the single best stylistic match to your reference standard across several topics. In her own words she refuses to oversimplify: on pipes, "It is actually a lot like that... That's not what it is, it is two file descriptors." That is exactly the analogy-that-is-also-the-real-mechanism approach you want.
- **Core Dumped** is a genuinely animated OS-internals channel, but YouTube blocks automated listing of its full catalog. I positively confirmed its process-creation video and several others (kernel boundaries, threads, scheduling, concurrency) but **could not confirm** dedicated Core Dumped videos on signals, file descriptors, or permissions.
- The strongest "why it's designed this way" sources are frequently the original author/designer writing directly: **Rob Landley** on the /usr split, **Lennart Poettering** on systemd/PID 1.
- Sysadmin "operational" topics (signals, env vars, disk exhaustion, sudo/su) are drowning in shallow how-to tutorials but lack a single analogy+mechanism+visual resource at the reference bar.

## Details - Topic by Topic

### 1. Processes, fork/exec, zombie/orphan processes
**Resource:** "The Weird Way Linux Creates Processes" - Core Dumped (animated YouTube video) - https://www.youtube.com/watch?v=SwIPOf2YAgI
**Why it matches:** Core Dumped animates what actually happens when you run a program - the fork-then-exec split, why Unix creates processes by cloning then replacing - rather than listing `ps`/`kill` syntax. It builds intuition visually then shows the real kernel mechanism, matching the analogy→mechanism arc. **Gap note:** this video centers on fork/exec; for the zombie/orphan lifecycle specifically (a terminated child stays in the process table until the parent `wait()`s and "reaps" it; orphans are re-parented to init/PID 1, which continuously reaps) the clearest accurate concise reference is the Wikipedia "Zombie process" article - accurate but not illustrated. Core Dumped's "Operating Systems Theory" playlist (https://www.youtube.com/playlist?list=PL9vTTBa7QaQPdvEuMTqS9McY-ieaweU8M) is the place to check for a dedicated process-lifecycle episode.

### 2. File descriptors and "everything is a file"
**Resource:** "File descriptors" - Julia Evans (illustrated comic) - https://wizardzines.com/comics/file-descriptors/ (also https://drawings.jvns.ca/file-descriptors/)
**Why it matches:** A near-perfect fit. The comic uses the analogy of the kernel handing your process integer tickets ("okay! that's file #7 for you"), then shows the real mechanism - stdin/stdout/stderr as fds 0/1/2, "read from stdin means read from file descriptor 0 (could be a pipe or file or terminal)," and sockets as files. Genuinely visual and technically accurate. Part of her "Bite Size Linux" material. Pair with her `lsof` comic to see fds live via `/proc/<pid>/fd`.

### 3. Linux permissions (rwx, UID/GID, setuid/setgid, sticky bit)
**Resource:** "Permissions" - Julia Evans (illustrated comic) - https://wizardzines.com/comics/permissions/ (also https://drawings.jvns.ca/permissions/)
**Why it matches:** Teaches the *model's logic*, not chmod syntax: it shows that file permissions are really 12 bits (setuid/setgid/sticky are the top three), how octal maps to binary (110 = rw- = 6, so 644 = rw-r--r--), and the reasoning for setuid via `ls -l /bin/ping` → `rws r-x r-x root root`, "the s means ping always runs as root." For the deeper reasoning behind setuid-on-executables, setgid-on-directories, and sticky-on-/tmp, pair it with the notes on her "Bite Size Linux" zine (well summarized at https://glumia.dev/blog/bite-size-linux/, which explains setuid lets limited users run trusted root programs like `passwd`, setgid-on-dirs fixes shared-workspace permissions, and sticky stops users deleting each other's files in /tmp).

### 4. Signals (SIGTERM, SIGKILL, SIGHUP, etc.)
**⚠️ HONEST FLAG - no single resource fully meets the bar.** I could not confirm an animated/illustrated explainer that does analogy + full mechanism (signal delivery, pending/blocked signal masks, handlers, why SIGKILL/SIGSTOP are uncatchable, the uninterruptible D-state) at the reference level.
**Best available:** The **GNU C Library manual, "Termination Signals"** chapter is the most accurate primary source on *why* each signal behaves as it does - e.g., SIGQUIT produces a core dump and deliberately should NOT delete temp files "so that the user can examine them in conjunction with the core dump," and SIGKILL "cannot be handled or ignored, and is therefore always fatal." URL: https://www.gnu.org/software/libc/manual/html_node/Termination-Signals.html . It is text-only reference material. Julia Evans' "Bite Size Linux" zine touches signals with drawings but is not a complete treatment. If you want video, check Jacob Sorber's C/OS channel and the Core Dumped OS-Theory playlist directly.

### 5. Environment variables and shell inheritance
**⚠️ HONEST FLAG - no single resource fully meets the bar.** Nearly all material is shallow "how to set an env var" tutorials. I found no single illustrated/animated resource that walks through the real mechanism - the `environ` array, how `execve()` passes it to the new program, and why a child gets a *copy* so it can never mutate its parent's environment (the reason `export` in a subshell doesn't affect the parent).
**Best available:** Julia Evans' "So You Want To Be A Wizard" and "Bite Size Command Line" zines cover the shell and inheritance visually but not exhaustively; Brian Ward's *How Linux Works, 3rd ed.* covers the environment/process-inheritance mechanism accurately in prose.

### 6. inodes, hard links vs symlinks
**Resource:** Julia Evans "Bite Size Linux" zine (illustrated) - https://wizardzines.com/zines/bite-size-linux/ - supplemented by "What is: hard link, symlink, and inode in Linux?" (RTFM) - https://rtfm.co.ua/en/what-is-hard-link-symlink-and-inode-in-linux/
**Why it matches:** Bite Size Linux is the illustrated option and treats inodes as a Linux fundamental. The RTFM article is the clearest mechanism walkthrough: via `ls -li` it shows a hard link is just a second directory entry pointing at the *same inode number* (link count increments to 2, 3, …), a symlink is a *separate* inode storing a path string, hard links can't cross filesystems (inode numbers are only unique within one FS), and it connects directly to the "No space left on device with free space showing" inode-exhaustion failure. Treated as a pair because no single resource is both perfectly visual and complete.

### 7. Filesystem Hierarchy Standard (why /etc, /var, /usr exist as they do)
**Resource:** "Understanding the bin, sbin, usr/bin, usr/sbin split" - Rob Landley (busybox mailing-list essay, Dec 9 2010) - https://lists.busybox.net/pipermail/busybox/2010-December/074114.html (illustrated PDF reprint: https://landley.net/writing/hackermonthly-issue022-pg33.pdf)
**Why it matches:** The definitive historical/design-reasoning piece. Landley shows the /usr split is a 1970s accident, verbatim: "around 1971 they upgraded to a PDP-11 with a pair of RK05 disk packs (1.5 megabytes each) for storage. When the operating system grew too big to fit on the first RK05 disk pack (their root filesystem) they let it leak into the second one, which is where all the user home directories lived (which is why the mount was called /usr)." His conclusion is blunt: the split "stopped making any sense before Linux was ever invented, for multiple reasons," calling it "a 1970's implementation detail that got carried forward for decades by bureaucrats who never question *why* they're doing things." Exactly the "why it works this way" depth you asked for - prose, not illustrated (though the Hacker Monthly reprint is laid out with graphics).

### 8. Disk and inode exhaustion - real failure mechanics
**⚠️ HONEST FLAG - no single analogy+mechanism+visual resource exists.**
**Best available (pair):** The **`core(5)` man page** enumerates precisely what silently breaks - a core dump is not produced when "The filesystem where the core dump file would be created is full; or has run out of inodes; or is mounted read-only; or the user has reached their quota for the filesystem" - https://man7.org/linux/man-pages/man5/core.5.html . For the classic "`df` shows free space but you still get *No space left on device*" inode-exhaustion mechanism (inodes are a fixed count set at filesystem creation; run out of them and no new file can be created even with free blocks), the RTFM article from topic 6 is the clearest. Combine the two; neither is illustrated. Brian Ward's *How Linux Works* also covers filesystem internals accurately.

### 9. systemd units, targets, and dependency ordering
**Resource:** "Rethinking PID 1" - Lennart Poettering (design essay, Apr 30 2010) - https://0pointer.de/blog/projects/systemd.html
**Why it matches:** Written by systemd's creator, this is the canonical "why it's designed this way." It frames the goal in one line - "For a fast and efficient boot-up two things are crucial: To start less. And to start more in parallel" - then explains units as typed, file-backed objects (service, mount, device, target, automount…), how dependencies (Requires/Conflicts) work, and how implicit ordering is derived (a mount of `/home/lennart` implicitly depends on the mount of `/home`; device/mount relationships where "the ordering and synchronization is done by the kernel, without any further management from userspace"). It's the reasoning, not a how-to - prose, not illustrated. **Runner-up:** the interactive "systemd by example" site if you prefer clickable diagrams over a design essay.

### 10. The Linux boot process end to end
**Resource:** "An introduction to the Linux boot and startup processes" - David Both, Opensource.com - https://opensource.com/article/17/2/linux-boot-and-startup - with Brian Ward, *How Linux Works, 3rd ed.* (No Starch, 2021) as the deeper reference.
**Why it matches:** The article walks firmware/POST → GRUB2 (stage 1 / 1.5 / 2) → kernel decompression + initramfs → systemd → targets, in order, and gives the *reasoning* for initramfs: mounting the real root filesystem needs drivers (LVM, RAID, encrypted, etc.) that live *on* that root filesystem, so a temporary RAM filesystem carries just enough to break the chicken-and-egg. Brian Ward's book is the connective, mechanism-first end-to-end treatment of the same journey. Neither is a whiteboard video, but both are accurate. (The ArchWiki "Arch boot process" page is the best free deep reference for the UEFI/EFI-stub path specifically.)

### 11. TCP/IP handshake, ports, and sockets
**Resource:** Julia Evans "Bite Size Networking!" zine (illustrated) - https://wizardzines.com/zines/bite-size-networking/
**Why it matches:** Illustrated, fundamentals-focused treatment of TCP, ports and sockets in her analogy+mechanism style, and the format you learn best from. **Runner-ups by need:** for the handshake/API mechanism at depth, *Beej's Guide to Network Programming* (the classic, thorough sockets tutorial); for a quick animated SYN/SYN-ACK/ACK view, ByteByteGo's short "How TCP Handshake Works" visualizations.

### 12. How a reverse proxy actually works under the hood
**Resource:** "The Architecture of NGINX" - Hussein Nasser (article) - https://medium.com/@hnasr/the-architecture-of-nginx-2b32fc0b7877
**Why it matches:** Genuinely mechanism-level: the master process vs worker processes, one worker per CPU core, and how workers accept connections - older versions had all workers contend on a shared listener socket, newer versions use `SO_REUSEPORT` socket sharding so the kernel load-balances across per-worker accept queues. His companion YouTube videos ("Proxy vs Reverse Proxy," "Layer 4 vs Layer 7 load balancing," and the HAProxy "Anatomy of a Request" talk) add the event-loop, buffering, and TLS-termination discussion (receive/send buffers via `net.ipv4.tcp_wmem`, `somaxconn` accept-queue limits). Best available for the connection-handling/event-loop depth requested; more diagram-and-talk than fully animated.

### 13. TLS/HTTPS handshake and certificate chains
**Resource (analogy + mechanism):** "How HTTPS Works" comic - https://howhttps.works/ - **paired with** Cloudflare's "What happens in a TLS handshake?" - https://www.cloudflare.com/learning/ssl/what-happens-in-a-tls-handshake/
**Why it matches:** howhttps.works is an explicitly comic-style explainer built on the "secret handshake" analogy that also covers certificate validation; Cloudflare's article gives the accurate step sequence (client hello with supported ciphers + client random → server hello + certificate + server random → client authenticates the cert against the issuing CA's signature → key exchange → both sides derive matching session keys) and clearly separates the asymmetric handshake from the symmetric session encryption that follows. **Deep runner-up:** "The Illustrated TLS 1.3 Connection: Every Byte Explained" - https://tls13.xargs.org/ - annotates every byte of a real handshake (including why TLS 1.3 disguises itself as TLS 1.2 for middlebox compatibility); unmatched for accuracy, though advanced.

### 14. SSH key exchange and authentication flow
**Resource:** "Deep Dive - How Does SSH Really Work?" - tusharf5 (illustrated blog) - https://tusharf5.com/posts/ssh-deep-dive/
**Why it matches:** Uses the client/server analogy then walks the real two-stage mechanism: (1) a Diffie-Hellman-style key exchange where each side independently computes the *same* shared symmetric key that is never transmitted, then (2) asymmetric public-key authentication of the client (server challenges, client signs with its private key, server verifies against `authorized_keys`), plus per-packet integrity hashes. It explicitly corrects the common misconception that the SSH key pair encrypts the whole session - it's only for authentication. **Runner-up for the certificate model (which most SSH content ignores):** smallstep's "If you're not using SSH certificates you're doing SSH wrong" - https://smallstep.com/blog/use-ssh-certificates/ .

### 15. PAM (Pluggable Authentication Modules) - what it is and WHY
**Resource:** "An introduction to Pluggable Authentication Modules (PAM) in Linux" - Susan Lauber, Red Hat blog - https://www.redhat.com/en/blog/pluggable-authentication-modules-pam (companion: "Anatomy of a PAM configuration file" - https://www.redhat.com/en/blog/pam-configuration-file)
**Why it matches:** Directly answers "why it's designed this way": historically every program rolled its own authentication, so PAM (originating from Sun's Solaris, ~1997) centralizes auth behind a single documented library. That modular architecture means developers never reinvent auth, and admins can swap or stack auth sources (Kerberos, SSSD, NIS, local files) and prioritize them per-service via `/etc/pam.d`. The companion post explains the account/auth/password/session module types, stacking, and control flags. Accurate, config-driven text - not illustrated.

### 16. sudo vs su and the Linux privilege model
**⚠️ HONEST FLAG - no single resource meets the bar.** Most content is command cheatsheets. I found no single analogy+mechanism+visual resource explaining the real privilege model (real vs effective UID, how `su` spawns a shell as another user vs how `sudo` consults `/etc/sudoers`, enforces policy, and logs, all built on setuid-root binaries).
**Best available (pair):** the setuid explanation in **Julia Evans' permissions comic** (topic 3) shows the exact underlying mechanism - a setuid-root binary like `ping` runs with its owner's (root's) privilege regardless of who launches it, which is precisely how `sudo`/`su` gain privilege - then read the `sudo` and `sudoers` man pages for policy/logging specifics. No polished single explainer identified; Brian Ward's *How Linux Works* covers the privilege model accurately in one place.

### 17. Cron internals and why race conditions happen
**Resource:** "Prevent cronjobs from overlapping in Linux" - Mattias Geniar (ma.ttias.be) - https://ma.ttias.be/prevent-cronjobs-from-overlapping-in-linux/
**Why it matches:** Explains the actual failure mechanism - a job scheduled every minute that takes longer than a minute starts overlapping itself, and if the jobs touch shared data/DB you get corruption plus a load-spiral vicious circle - and the real fix: `flock` advisory locks, where "the kernel releases the lock automatically when the process exits, so there is no cleanup to do" (unlike a hand-rolled PID lockfile that goes stale on `kill -9` or power loss). Published 2012, "Updated Jun 6, 2026" with a correctness note that `flock` ships in `util-linux` (there is no separate `flock` package). Mechanism-first and accurate; it's a blog post rather than an illustrated explainer, and it covers the locking/race angle more than cron's internal scheduling implementation - so it's the best available rather than a flawless match. For the "why systemd timers avoid this" angle, note timers won't start a second instance if one is still running, so `flock` isn't needed there.
